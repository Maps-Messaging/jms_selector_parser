# CLAUDE.md

This file provides guidance for AI assistants working with the JMS Selector Parser codebase.

## Project Overview

An extensible JMS Selector Parser library that conforms to JMS Selector Syntax (a subset of SQL92). It parses selector expressions and evaluates them against Java objects for filtering. Part of the MapsMessaging ecosystem but usable standalone.

- **Group/Artifact:** `io.mapsmessaging:jms_selector_parser`
- **License:** Apache 2.0 with Commons Clause
- **Java Version:** 17 (standard profile), 21 (ml profile)

## Build Commands

```bash
# Standard build (default profile, Java 17)
mvn -Dgpg.skip=true clean install -Pstandard

# ML build (Java 21 required, includes ML algorithms)
mvn -Dgpg.skip=true clean install -Pml

# Run tests only
mvn -Dgpg.skip=true test -Pstandard

# Run a single test class
mvn -Dgpg.skip=true test -Pstandard -Dtest=SelectorConformanceTest

# Run a single test method
mvn -Dgpg.skip=true test -Pstandard -Dtest=SelectorConformanceTest#testMethod

# Generate coverage report (JaCoCo)
mvn -Dgpg.skip=true verify -Pstandard
```

**Important:** Always pass `-Dgpg.skip=true` for local builds to skip GPG signing. The CI pipeline uses the `standard` profile by default.

## Architecture

### Core Flow

1. **Parse:** `SelectorParser.compile(String)` parses a JMS selector string into an `Operation` AST
2. **Wrap:** The `Operation` is wrapped in a `ParserOperationExecutor` implementing `ParserExecutor`
3. **Resolve:** `EvaluatorFactory.create(Object)` creates the right `IdentifierResolver` for the target object type
4. **Evaluate:** `ParserExecutor.evaluate(Object)` resolves identifiers and walks the AST to produce a boolean result

### Key Interfaces

| Interface | Purpose |
|-----------|---------|
| `ParserExecutor` | Compiled selector; call `evaluate(Object)` to filter objects |
| `IdentifierResolver` | Read-only lookup: `get(String key)` returns field values |
| `IdentifierMutator` | Extends `IdentifierResolver` with `set()` and `remove()` |
| `ParserExtension` | SPI for registering custom parser functions via ServiceLoader |

### Operator Hierarchy (Composite pattern)

```
Operation (abstract)
├── ComputableOperator
│   ├── ArithmeticOperator → Add, Subtract, Multiply, Divide
│   └── ComparisonOperator → Equal, Different, GreaterThan, LessThan, etc.
├── LogicalOperator → And, Or, Not
├── FunctionOperator → Between, In, Is, Like
└── Action → Set, Remove
```

### Object Type Resolution

`EvaluatorFactory.create(Object)` detects the type in this priority order:
1. `IdentifierResolver` (already implements interface) - pass through
2. `Map` - wraps in `MapEvaluator`
3. `JsonObject` (Gson) - wraps in `JsonEvaluator`
4. Anything else - wraps in `BeanEvaluator` (reflection-based)

### Extension System

Custom functions are loaded via `java.util.ServiceLoader`. Implementations of `ParserExtension` are registered in `META-INF/services/io.mapsmessaging.selector.extensions.ParserExtension`. The `ParserFactory` loads all extensions at startup.

## Code Conventions

### Style

- **Indentation:** 2 spaces in JavaCC grammar files, standard Java indentation in Java files
- **Braces:** Opening brace on same line (K&R style)
- **Encoding:** UTF-8 throughout
- **Line length:** ~100 characters typical

### Naming

- **Classes:** PascalCase (`ParserExecutor`, `AndOperator`)
- **Methods:** camelCase (`evaluate()`, `compile()`)
- **Constants:** UPPER_SNAKE_CASE (`MULTI_CHARACTER`)
- **Packages:** lowercase (`io.mapsmessaging.selector.operators.arithmetic`)
- **Test classes:** Suffixed with `Test` (e.g., `AdditionTest`, `SelectorConformanceTest`)

### Patterns Used

- **Composite:** `Operation` tree for expression evaluation
- **Strategy:** `IdentifierResolver` implementations for different object types
- **Factory:** `EvaluatorFactory` for type detection and resolver creation
- **ServiceLoader:** Extension mechanism for custom parser functions
- **Template Method:** `ComparisonOperator` defines abstract `compare()` for subclasses
- **Caching:** `BeanEvaluator` caches reflection metadata (PropertyDescriptors)

### Lombok

The project uses Lombok annotations (`@Getter`, `@Setter`, `@Data`) to reduce boilerplate. The `lombok` dependency is `provided` scope and processed via annotation processor in `maven-compiler-plugin`.

### Type Handling

- Operators work with `Object` operands (heterogeneous typing)
- Automatic `String` to `Number` parsing via `parseStringToNumber()` in `Operation`
- Number hierarchy: operations handle `Double`, `Float`, `Long` conversions
- Null-safe evaluation throughout; missing identifiers return `null`

## JavaCC Grammar

The parser is generated from JavaCC `.jj` grammar files. **Do not edit `SelectorParser.java` directly** - it is generated during the build. To modify parser behavior:

1. Edit the grammar file in `src/main/javacc/standard/SelectorParser.jj` (or `ml/SelectorParser.jj` for ML profile)
2. Run `mvn generate-sources` to regenerate the parser
3. The generated parser appears in `target/generated-sources/javacc/`

The `ActionParser.jj` defines grammar for SET/REMOVE mutation actions.

## Common Development Tasks

### Adding a New Operator

1. Create a new class extending the appropriate base (`ArithmeticOperator`, `ComparisonOperator`, `LogicalOperator`, or `FunctionOperator`)
2. Implement `evaluate(IdentifierResolver)` and `compile()` methods
3. Wire it into the JavaCC grammar (`SelectorParser.jj`)
4. Add corresponding tests in the matching test package
5. Regenerate the parser with `mvn generate-sources`

### Adding a New Object Type Resolver

1. Create a class implementing `IdentifierResolver` (or `IdentifierMutator` for read-write)
2. Add detection logic to `EvaluatorFactory.create(Object)`
3. Add integration tests

### Adding a New ML Algorithm (ml profile)

1. Create operation class in `src/main/java-ml/.../operators/functions/ml/impl/functions/`
2. Extend `AbstractMLModelOperation` (labeled data) or `RawDataMLModelOperation` (unsupervised)
3. Create corresponding function classes for each query type
4. Register in the ML grammar (`src/main/javacc/ml/SelectorParser.jj`)
5. Add tests in `src/test/java-ml/`

## Important Notes

- The `extentions` package name (under `operators`) is a historical typo; do not rename it as it would break existing APIs
- `SelectorParser.java` in the main source tree is generated by JavaCC - always edit the `.jj` grammar instead
- The CI strips the `ml-` prefix from the version in `pom.xml` before deploying (see `.buildkite/pipeline.yml`)
- The `BeanEvaluator` uses Java reflection and caches `PropertyDescriptor` lookups for performance
- Bean property names are case-sensitive in selector expressions
